How Ledger works

Ledger reads a hand-maintained calls.json — one entry per published rating, each tracing back to an actual report — and a daily Python job fills in the rest:

  1. Return since call — (live_price / price_at_call − 1) × 100. This is a price return: dividends are excluded, so income-paying names (HDFC Bank, Coca-Cola, Visa…) will understate total return versus what a holder actually earned.
  2. Alpha — the call's return since call date, minus its benchmark's return over the identical window. India calls are measured in INR against the Nifty 50; US calls in USD against the S&P 500 — never mixed across currencies.
  3. Distance to target — (target / live_price − 1) × 100. Positive means the target is still above the live price; negative means the live price has moved past it.
  4. Hit / miss (closed calls only) — a call closes when a later report revises it (e.g. Infosys HOLD → ACCUMULATE, ICICI HOLD → REDUCE). Its window runs from its own call date to the date of the revision, using the revision's price as the closing price. It's marked a hit if the price ever crossed the target during that window (reached it on the way up for an upside thesis, or fell to it for a downside/fair-value thesis) — otherwise a miss.

Where a call was revised, both the closed original and the open revision are kept — nothing is deleted from the record. The summary header's hit rate, average alpha, and best/worst call are computed across every row shown here, open and closed alike.
